Problem & Approach

Manual Oversight at Scale

Nassau manages 120K+ insurance policies through third-party administrators responsible for transactions, claims, disbursements, and valuations. Compliance review was fully manual: analysts pulled reports from disparate systems, reconciled entity mismatches by hand, and scored quality inconsistently across 8 activity types, with no historical trend data and no self-serve path.

I was brought in as the sole developer to build the system end to end. The binding constraint was that compliance analysts aren't engineers, so the solution had to be entirely self-serve. That ruled out ad-hoc SQL and pointed toward R Shiny as the delivery layer, with parquet-backed storage via Apache Arrow for persistent, auditable records.

For scoring, I designed an activity-specific weighted system rather than a single global score. Disbursement reviews carry different risk profiles than routine transaction checks, so weights and pass thresholds are defined per activity type and externalized as config, meaning the team can tune the system as their understanding matures without a code deploy.

1. ETL Pipeline

The pipeline (processing_data.Rmd) pulls from 7 insurance activity streams in the AWS Enterprise Data Warehouse: transactions, claims, disbursements, policies, automated activities, and annuity/life valuations. Each source has its own schema and update cadence, requiring activity-specific ingestion logic.

  • ▸Monthly delta processing: only new or updated records are pulled each run
  • ▸Many-to-many entity deduplication across roles, names, and agent IDs using latest-record logic
  • ▸Output written to parquet via Apache Arrow for fast reads and audit trail persistence
  • ▸Handles 1M–100M row datasets across the full policy book

2. Compliance Scoring Engine

The scoring engine evaluates each policy review against activity-specific weighted checklists. Rather than a one-size-fits-all quality score, each of the 8 activity types (transactions, claims, disbursements, automated activities, annuity values, life values, and two others) has its own category weights and configurable pass threshold, defaulting to 80%.

  • ▸Weighted scoring per activity type: high-risk activities carry greater weight toward overall compliance score
  • ▸Dollar-risk classification (High / Low / Normal) applied via configurable threshold rules
  • ▸Pass/fail determination at configurable threshold, defaulting to 80%
  • ▸12-month rolling quality trend computed per TPA for longitudinal oversight

3. R Shiny Dashboard

The dashboard (insurance_oversight_app.R) is the primary interface for the 30-person compliance team. It was built for self-serve use: analysts can flag reviews, adjust scoring thresholds, and track policy status without any engineering support.

  • ▸Executive KPI panels: contract counts, completion rates, risk breakdowns by TPA
  • ▸Activity-level quality assessment with drill-down per policy
  • ▸Risk tier classification (High / Low / Normal) with exception flagging
  • ▸12-month trend charts via Plotly and ggplot2
  • ▸Interactive data tables (DT) for analyst-level review and export
  • ▸Reactive programming: all views update instantly on filter changes
If I Were to Revisit It

What Scales, What Doesn't

It was designed to scale within a bounded compliance workflow: parquet storage, configurable thresholds, and modular ETL all hold up well. The main architectural tradeoff I'd revisit is the parquet upsert pattern for review persistence, which would need a proper indexed database at higher review volumes.

What scales well

  • ▸Parquet storage: columnar format handles wide, sparse insurance data efficiently and reads far faster than CSVs at large row counts
  • ▸Externalized config: thresholds like HIGH_THRESHOLD and quality_pass_threshold adapt to new business rules without code changes
  • ▸Contract sampling design: pulling 50 contracts per month per stream keeps memory usage bounded regardless of warehouse growth
  • ▸Modular ETL: each activity stream is processed independently, so adding a new stream is additive rather than a refactor

What doesn't

  • ▸save_review_to_parquet(): reads the full file, modifies in memory, rewrites the whole thing. Fine at a few hundred reviews; an O(n) bottleneck at tens of thousands. Indexed upserts in a real database would replace this
  • ▸collect() everywhere: every stream is fully pulled into R memory before filtering. Works at current volumes; breaks under production-scale row counts without lazy evaluation or server-side filtering
  • ▸Hard-coded checklists: adding a new insurance product type requires a code deploy, not a config change. A database-driven checklist system would be more extensible
  • ▸Single-threaded Shiny: handles a 30-person team fine; for 300 concurrent users it would need async processing or a different framework
