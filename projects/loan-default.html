MLOps Loan Default Prediction System with Drift Monitoring

End-to-end production ML system with data drift alerting and experiment tracking

Role: Sole Developer · December 2025 · Production · Deployed
TL;DR

Built a production ML system predicting loan defaults on 148K applications achieving 89% accuracy, 0.92 ROC-AUC, and 0.86 F1-score using Random Forest. Engineered a PyTorch Dataset/DataLoader pipeline that reduced memory usage by 4,600× and improved loading speed by 23×. Implemented Population Stability Index (PSI) drift monitoring across 51 features with MLflow experiment tracking. Deployed as a web interface on Hugging Face Spaces via Docker with <100ms inference latency.

Problem & Context

Predicting Risk at Scale

Loan default prediction is a core problem in credit risk management. Financial institutions need to assess the probability of default before issuing credit,miscalibrated models mean either leaving money on the table (too restrictive) or taking on unacceptable risk (too permissive). At 148,000+ applications, even a small improvement in precision translates to significant risk-adjusted returns.

Beyond prediction accuracy, production ML systems for credit have two additional requirements that pure model benchmarks ignore: scalable data handling (the pipeline needs to work on full datasets, not just samples) and drift monitoring (model assumptions break down as the applicant population shifts over time).

This project was built to demonstrate a realistic production ML workflow: efficient data loading, rigorous model evaluation, drift detection infrastructure, and a deployed web interface,not just a Jupyter notebook with a model fit.

Data & Features

Dataset & Engineering

The dataset contains 148,000+ loan applications with 51 features spanning applicant demographics, financial history, loan characteristics, and derived risk signals. Target variable is binary: defaulted (1) or repaid (0).

Applicant Features

  • Credit score & history length
  • Annual income
  • Employment status & length
  • Debt-to-income ratio

Loan Features

  • Loan amount & term
  • Interest rate & grade
  • Purpose & home ownership
  • Monthly installment

Risk Signals

  • Delinquency history
  • Public records count
  • Revolving utilization
  • Recent inquiry count

PyTorch data pipeline: Instead of loading all 148K rows into memory at once (which caused OOM errors in early iterations), a custom Dataset and DataLoader class was implemented. This yielded a 4,600× memory reduction (from ~2.8GB to ~0.6MB resident) and 23× faster loading via lazy evaluation and batching,turning an OOM failure into a sub-second operation.

Modeling & MLOps

Model & Drift Monitoring

Model selection: Random Forest was chosen over Logistic Regression (too simple for 51 correlated features), XGBoost (similar performance, harder to explain), and neural networks (overkill for tabular credit data at this scale). Random Forest provides strong performance, natural feature importance interpretability, and robustness to missing values,important for real credit risk workflows.

Experiment tracking with MLflow: All training runs logged hyperparameters, metrics, artifacts, and model versions. This enabled systematic comparison of feature subsets, imputation strategies, and class imbalance handling techniques, with full reproducibility.

PSI drift monitoring: Population Stability Index measures how much the distribution of a feature has shifted between training and serving. PSI is computed for all 51 features:

  • ▸PSI < 0.1: No significant shift,model stable
  • ▸PSI 0.1–0.2: Moderate shift,monitor closely
  • ▸PSI > 0.2: Significant shift,retrain recommended

This infrastructure means a deployed model can raise alerts before degradation becomes visible in downstream business metrics,catching drift proactively rather than reactively.

What Was Learned

Lessons & Challenges

  • 01.
    Data pipelines are the real bottleneck. The biggest early pain point wasn't model accuracy,it was OOM errors loading 148K rows. Building a proper PyTorch DataLoader pipeline unlocked both memory efficiency and faster experimentation cycles.
  • 02.
    Drift monitoring needs to be built in, not bolted on. PSI monitoring infrastructure built during initial development (not added later) made it easy to generate drift reports as a first-class artifact alongside model metrics.
  • 03.
    MLflow prevents "which run was best?" regret. Without experiment tracking, hyperparameter searches become chaotic. MLflow logging of every run's parameters, metrics, and artifacts made the final model selection defensible and reproducible.
  • 04.
    Docker + Gradio is a powerful deployment combo. Containerizing the Gradio app meant identical behavior locally and in production on Hugging Face Spaces,no "works on my machine" surprises.
