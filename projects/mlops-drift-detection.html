Problem & Approach

A Model That Ships Is Not a Model That's Done

Most ML projects end at the test set. A model scores well on held-out data, gets deployed, and the work is considered finished. The problem is that loan applicant data doesn't stay static, when changes in economic conditions, or new products launch, and even applicant demographics change, a model trained on last year's data will silently degrade as the real-world distribution drifts away from what it learned, with no obvious signal that anything is wrong. This project builds both the prediction system and the monitoring layer, treating deployment as the beginning of the problem rather than the end.

Three detection methods cover different types of drift. PSI (Population Stability Index) flags when continuous feature distributions shift beyond a threshold relative to the training baseline. A KS test catches finer distribution changes that PSI might miss at lower drift magnitudes. Chi-squared handles categorical features where PSI doesn't apply. Each batch of predictions is checked against baseline statistics computed from the training set, then the drift status is run through the API, so consumers can decide how much to trust the current predictions.

Dataset

148K Loan Applications, 51 Features

Found a real dataset via Kaggle with 148,670 loan applications. I engineered 51 features spanning credit history, income ratios, loan terms, payment behavior, and delinquency indicators. Standard stratified 80/20 train/test split. Data cleaning was done through a Human-in-the-Loop process with various AI Agents, and feature engineering was the majority of the preprocessing work, building interaction terms and ratio features that raw loan data doesn't include directly.

Models & Results

Three Models Evaluated, One Deployed

Random Forest, XGBoost, and a PyTorch neural net were all evaluated. Random Forest was selected for deployment based on the combination of predictive performance and interpretability.

  • Random Forest (selected). ROC-AUC 0.92, F1 0.86, 88.93% accuracy. Deployed via FastAPI.
  • XGBoost. Gradient boosting on tabular loan data is a natural fit but offered no meaningful advantage over RF.
  • PyTorch neural net. Baseline to see if deep learning added anything on structured tabular data. It did not.
All experiments tracked in MLflow, including hyperparameter configs, evaluation metrics, and model artifacts. This made it straightforward to compare runs and roll back to earlier model versions.
API & Deployment

FastAPI, Docker, Sub-100ms Inference

The FastAPI application wraps the trained model with four endpoints. (1) single prediction, (2) batch prediction, (3) drift status check, and (4) on-demand drift detection. Predictions are logged to a SQLite database via SQLAlchemy, which makes it possible to run drift analysis against stored prediction history rather than just live input batches. The full system is containerized with Docker, and OpenAPI documentation is generated automatically.

Removed the live demo because HuggingFace Spaces was experiencing free-tier cold start latency that made the sub-100ms claim meaningless to anyone actually clicking through. The GitHub repo has setup instructions and the Docker image runs locally without issue.

WHAT I LEARNED

Building the drift monitoring layer changed how I think about model deployment. The prediction API was the straightforward part. Deciding what counts as meaningful drift, what thresholds should trigger an alert, and how to communicate model uncertainty back to whoever is consuming the predictions turned out to be a more interesting problem than the modeling work itself. The bigger takeaway is that a model isn't a finished artifact when it ships. It's a liability that needs watching.
