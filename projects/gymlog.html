Problem & Approach

Real Application, Not a Demo

Fitness tracking apps either give you a bloated, subscription-locked experience (MyFitnessPal, Whoop) or a barebones spreadsheet. Neither integrates the three core pillars of training in one place: workout execution, nutrition logging, and progressive overload guidance informed by your actual history.

GymLog was built as both a personal tool and a full-stack engineering project, the goal was to build something production-quality with a real ML integration, not a demo. This meant handling real data entry patterns (inconsistent exercise names, missing fields, browser crashes mid-workout), designing a normalized schema that could evolve, and deploying a system that non-technical users could actually use.

Why Flask Over Django

Flask was chosen because the project required full control over API design, 40+ endpoints built exactly to spec, and didn't need Django's built-in admin or form-handling machinery. The ML component was scoped to Gradient Boosting Regression rather than a neural network because user-specific training history is sparse: most users have 10 to 50 workouts per exercise, making a network's generalization advantage irrelevant while GBR's explicit 12-feature structure handles sparse histories reliably. Deployment on Gunicorn with PostgreSQL in production (SQLite in development) was a deliberate choice to treat this as a real application from day one.

Key Design Decisions

  • GBR over neural network: user training histories are sparse (10–50 workouts per exercise); GBR's 12-feature explicit structure handles sparsity with no generalization cost
  • Pickle model persistence: one model file per exercise-user pair; lightweight, retrainable on demand without a versioned model registry at this scale
  • Dual-API fallback chain: OpenFoodFacts has excellent US packaged food coverage but gaps in international products; USDA FoodData Central fills those gaps with a custom relevance scorer resolving conflicts
  • Draft auto-save as first-class feature: is_draft=True on every keystroke, excluded from history views, converted to completed record on "Finish Workout", 100% data retention with no UX complexity
  • PostgreSQL/SQLite parity: production uses PostgreSQL via DATABASE_URL env var, development uses SQLite; same ORM layer, zero code changes between environments
Notable Implementations

Six Technical Highlights

1. Progressive Overload ML Model ml_models/progressive_overload.py

A Gradient Boosting Regressor (100 estimators, depth 4) trained on per-user workout history to predict the optimal weight for each exercise next session. 12 engineered features: last 5 workout weights, days since last workout, total training volume, success rate, progression velocity, and rep range analysis. Outputs include confidence intervals (±5%), smart rounding to the nearest 2.5 lb plate increment, and model versioning for continuous learning. Falls back to +2.5 lbs heuristic for exercises with fewer than 5 logged sessions.

2. Dual-API Food Search with Relevance Scoring app.py

Single-API nutrition data is insufficient. The system implements an intelligent fallback chain: primary OpenFoodFacts (15+ results), fallback to USDA FoodData Central. A custom relevance algorithm ranks results by: exact phrase matching (+50 pts), complete word presence (+30 pts), word boundary detection, and length penalization. Result: 40%+ improvement in food search relevance over single-API approach.

3. Exercise Name Normalization System utils.py

Users enter exercise names inconsistently, "DB bench", "dumbbell bench press", "bench DB", causing fragmented ML training data. A multi-stage normalization pipeline uses regex pattern matching, 40+ abbreviation expansions (DB→Dumbbell, RDL→Romanian Deadlift), word reordering, and special case handling to standardize 100+ exercise variations. Includes a preview-before-apply UI so users can confirm changes without silent correction.

4. Draft Workout Auto-Save System app.py

Browser crashes and accidental navigation were causing workout data loss. The solution: every edit auto-saves to is_draft=True. Drafts are excluded from history views but visible in the active session. A 24-hour cleanup task removes stale drafts. On "Finish Workout", the draft is converted to a completed record. Result: 100% data retention with no user-facing complexity.

5. Multi-Day Template Scheduling with Backwards Compatibility app.py

Original schema limited templates to one day of the week. The migration to multi-day support was zero-downtime: a TemplateSchedule association table for many-to-many scheduling was added while keeping the deprecated day_of_week field for existing data. The API accepts both single-day and array formats; queries check both fields. Clean backwards-compatible migration with no data loss.

6. Diagonal Split Calendar Visualization templates/index.html

When two workout templates fall on the same calendar day, the cell renders as a 135° diagonal gradient splitting both template colors 50/50. Dynamic CSS gradient generation detects 2+ templates on the same day, extracts hex colors, converts to rgba for opacity control, and generates the gradient inline. Hover shows a tooltip with both workout names. Elegant solution to a genuine UX problem that appeared only after real usage data emerged.

IF I WERE TO REVISIT IT

What Scales, What Doesn't

The application is production-quality and the deployed version works. The main constraints are design decisions that made sense for a solo-developer personal tool but would require re-architecture at team scale or under real user load.

What scales well

  • GBR feature engineering: the 12-feature structure for progressive overload generalizes naturally, adding new features is additive, not architectural. The explicit feature set also makes the model's predictions debuggable in a way that a neural network would not be.
  • Dual-API fallback with custom scoring: the relevance algorithm is content-agnostic. The same pattern, primary source, scored fallback, explicit tie-breaking, applies to any domain where two data sources have complementary strengths.
  • Draft auto-save as data integrity system: treating every user action as provisional until confirmed is the right abstraction for any data-entry application. The is_draft pattern is reusable.
  • Backwards-compatible schema migration: keeping the deprecated day_of_week field alongside the new TemplateSchedule table while accepting both formats in the API is the right way to handle breaking schema changes in a live system.

What doesn't

  • Schema design decisions compound: adding multi-day template support after launch required careful backwards compatibility work. Designing for extensibility upfront, even when it feels like over-engineering, saves painful migrations. The initial one-day-per-template constraint was not revisited early enough.
  • User data consistency requires automation: asking users to normalize exercise names themselves doesn't work, they won't. Automated normalization with a preview step (not silent correction) was the right balance, but it should have been designed in from the start, not added reactively after training data fragmentation emerged.
  • Single-file monolith: app.py at 1,844 lines is the classic Flask monolith that works until it doesn't. At team scale, splitting by blueprint (auth, workouts, nutrition, ML) would be the first refactor, it's mechanical work that becomes load-bearing when multiple developers touch the same file.
  • No auto-save was a mistake: the first version had no auto-save. After losing a long workout once, building a proper draft system became the highest priority feature. Every data-entry application should have this from day one.
