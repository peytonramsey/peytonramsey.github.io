GymLog,Full-Stack Fitness Tracking App

Production-ready web application with ML-powered training recommendations

Role: Sole Developer · 2025 · ~11,500 lines of code · v1.0.4
TL;DR

GymLog is a production-ready full-stack fitness tracking web application built with Python Flask and PostgreSQL,11,500+ lines of code, 40+ REST endpoints, 12 database models, and 6 Alembic migration versions. Features include workout logging with draft auto-save, a Gradient Boosting Regressor for progressive overload predictions, dual-API nutrition search with custom relevance scoring, barcode food scanning, and a glassmorphism UI with Chart.js visualizations. Deployed live with Gunicorn on cloud infrastructure.

Problem & Context

Why Build This?

Fitness tracking apps either give you a bloated, subscription-locked experience (MyFitnessPal, Whoop) or a barebones spreadsheet. Neither integrates the three core pillars of training in one place: workout execution, nutrition logging, and progressive overload guidance informed by your actual history.

GymLog was built as both a personal tool and a full-stack engineering project,the goal was to build something production-quality with a real ML integration, not a demo. This meant handling real data entry patterns (inconsistent exercise names, missing fields, browser crashes mid-workout), designing a normalized schema that could evolve, and deploying a system that non-technical users could actually use.

Notable Implementations

Six Technical Highlights

1. Progressive Overload ML Model ml_models/progressive_overload.py

A Gradient Boosting Regressor (100 estimators, depth 4) trained on per-user workout history to predict the optimal weight for each exercise next session. 12 engineered features:

  • ▸Temporal: last 5 workout weights
  • ▸Recovery: days since last workout
  • ▸Volume: total training load
  • ▸Performance: success rate, progression velocity
  • ▸Rep range analysis
  • ▸Fallback: +2.5 lbs heuristic for new exercises

Outputs include confidence intervals (±5%), smart rounding to nearest 2.5 lb plate increment, and model versioning for continuous learning.

2. Dual-API Food Search with Relevance Scoring app.py

Single-API nutrition data is insufficient. The system implements an intelligent fallback chain: primary OpenFoodFacts (15+ results), fallback to USDA FoodData Central. A custom relevance algorithm ranks results by:

  • ▸Exact phrase matching (+50 pts)
  • ▸Complete word presence (+30 pts)
  • ▸Word boundary detection and length penalization

Result: 40%+ improvement in food search relevance over single-API approach.

3. Exercise Name Normalization System utils.py

Users enter exercise names inconsistently ("DB bench", "dumbbell bench press", "bench DB") causing fragmented ML training data. A multi-stage normalization pipeline uses regex pattern matching, 40+ abbreviation expansions (DB→Dumbbell, RDL→Romanian Deadlift), word reordering, and special case handling to standardize 100+ exercise variations. Includes a preview-before-apply UI so users can confirm changes.

4. Draft Workout Auto-Save System app.py

Browser crashes and accidental navigation were causing workout data loss. The solution: every edit auto-saves to is_draft=True. Drafts are excluded from history views but visible in the active session. A 24-hour cleanup task removes stale drafts. On "Finish Workout", the draft is converted to a completed record. Result: 100% data retention with no user-facing complexity.

5. Multi-Day Template Scheduling with Backwards Compatibility app.py

Original schema limited templates to one day of the week. The migration to multi-day support needed to be zero-downtime: a TemplateSchedule association table for many-to-many scheduling was added while keeping the deprecated day_of_week field for existing data. The API accepts both single day and array formats; queries check both fields. Clean backwards-compatible migration.

6. Diagonal Split Calendar Visualization templates/index.html

When two workout templates fall on the same calendar day, the cell renders as a 135° diagonal gradient splitting both template colors 50/50. Dynamic CSS gradient generation detects 2+ templates on the same day, extracts hex colors, converts to rgba for opacity control, and generates the gradient inline. Hover shows a tooltip with both workout names. Elegant solution to a real UX problem.

What Was Learned

Lessons & Challenges

  • 01.
    Schema design decisions compound. Adding multi-day template support after launch required careful backwards compatibility work. Designing for extensibility upfront (even when it feels like over-engineering) saves painful migrations.
  • 02.
    User data consistency requires automation. Asking users to normalize exercise names themselves doesn't work,they won't. Automated normalization with a preview step (not silent correction) was the right balance of control and convenience.
  • 03.
    Single-API data sources are fragile. OpenFoodFacts has excellent coverage for US packaged foods but gaps in international products. The fallback to USDA plus custom relevance scoring solved what would have been a persistent user pain point.
  • 04.
    Auto-save is not optional in data entry apps. The first version had no auto-save. After losing a long workout once, building a proper draft system became the highest priority feature,and made the whole UX feel professional.
