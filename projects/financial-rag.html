Financial News RAG Analyst

An intelligent information retrieval system for financial markets

Role: Sole Developer · January 2026 · Production · Deployed
TL;DR

Built a production RAG system that enables natural-language queries across 500+ financial articles, reducing research time from 30+ minutes to under 5 seconds. The system achieves 92% retrieval precision and 90% answer accuracy across 30 test queries, with FinBERT-powered sentiment analysis enriching every response. Deployed on Hugging Face Spaces via Docker with end-to-end latency of 2.5–4.5 seconds.

Problem & Context

The Problem with Financial News

Financial markets generate approximately 10,000+ news articles daily across outlets like Bloomberg, Reuters, the Wall Street Journal, and MarketWatch. Analysts, portfolio managers, and traders need to rapidly synthesize this information to answer questions like "What are the current risks to tech stocks?" or "How is the Fed addressing inflation?"

Traditional approaches are broken: manual research takes 30+ minutes per query, keyword search requires knowing exactly what to look for (5–10 minutes), and professional terminals like Bloomberg or FactSet cost thousands per seat. Generic LLMs hallucinate financial facts and have knowledge cutoffs. News aggregators can surface articles but can't synthesize or reason across them.

The goal was to bridge this gap,combining the semantic understanding of LLMs with real, grounded news content to deliver fast, accurate, cited answers to financial questions.

Data & Indexing

Building the Knowledge Base

Financial news is sourced from the Finnhub API, covering general market news and company-specific articles for 50 priority tickers across 6 sectors:

  • ▸Technology 25% (AAPL, MSFT, NVDA, META, GOOGL)
  • ▸Financial 25% (JPM, BAC, GS, V, MA)
  • ▸General Market 20% (SPY, Fed policy, macro news)
  • ▸Healthcare 12%, Consumer 10%, Energy 8%

Raw articles go through a 5-stage filtering pipeline that reduces 500+ raw articles down to 50 high-quality, diverse articles (90% noise reduction):

Stage What It Does Remaining
1. Quality Min 100-char content, 15-char title; blocks PR wires & spam keywords ~287
2. Relevance Retains articles mentioning priority tickers or macro topics ~156
3. Deduplication Removes exact & fuzzy duplicates (SequenceMatcher, threshold=0.75) ~98
4. Quality Score Ranks by recency, length, source tier; premium sources +30 pts ~98
5. Diversity Sector quotas enforced (e.g., max 25 Technology articles) 50

Each article is then embedded using all-MiniLM-L6-v2 (384-dimensional), split into chunks of 1,000 characters with 100-character overlap via RecursiveCharacterTextSplitter, yielding ~120 chunks. Each chunk retains metadata: title, source, date, URL, FinBERT sentiment label/score, and sector. The index is stored in ChromaDB (~15 MB).

Key Component Decisions

  • ▸all-MiniLM-L6-v2 for embeddings,fast CPU inference (~10ms/chunk) with strong semantic similarity at only 384 dimensions
  • ▸ChromaDB for vector storage,persistent, local, supports cosine similarity with metadata filtering, ~15MB index for 120 chunks
  • ▸FinBERT (ProsusAI) for sentiment,domain-specific model trained on financial texts; 86% accuracy vs ~70% for generic BERT
  • ▸Dual LLM backends,Llama-3.1-8B-Instruct via HuggingFace Inference API (2–4s) for production; Phi-3 Mini via Ollama (5–10s) for local dev
  • ▸LangChain orchestration,structured prompt templates, context assembly, source attribution, streaming token output
Evaluation & A/B Testing

Measuring What Matters

Rather than relying on "it feels good" evaluation, the system was tested rigorously across retrieval quality, generation quality, and latency.

A/B test on article window size: tested 25, 50, 75, and 100 articles in the index. The 100-article window provided the optimal balance,more coverage without significantly increasing latency or degrading retrieval precision from irrelevant chunks diluting results. Below 50 articles, precision dropped for niche sector queries.

Retrieval evaluation: 20 diverse queries manually evaluated. Top-5 results contained a relevant article 92% of the time; highly relevant 78% of the time.

Sentiment validation: 50 articles manually labeled by human annotators. FinBERT achieved 86% overall agreement, 88% positive/negative precision, 82% neutral recall. Neutral class is hardest,financial news often has mixed signals.

What Was Learned

Lessons & Challenges

  • 01.
    Multi-stage filtering is critical. Raw financial APIs are full of press releases, duplicates, and low-quality aggregator content. The 90% noise reduction from 500+ to 50 articles dramatically improved answer quality,more isn't always better in RAG.
  • 02.
    Domain-specific models matter. FinBERT's 86% accuracy vs ~70% for generic BERT isn't just a number,financial language has domain-specific signals ("beat expectations" = positive) that generic models miss entirely.
  • 03.
    Evaluation discipline prevents overconfidence. Manual evaluation on 20–50 samples instead of "it feels good" revealed that neutral sentiment recall (82%) was the weak point,sentiment is often mixed in financial news and article-level labels don't always reflect query-specific sentiment.
  • 04.
    Dual backend flexibility has real value. Supporting Ollama locally (no API cost) and HuggingFace cloud (production) via a single environment variable meant fast iteration during development without burning API credits.
  • 05.
    Static knowledge base is a real limitation. The system requires periodic re-ingestion for fresh news. Real-time updates would require webhook integration or a continuous ingestion pipeline,a natural next step.
