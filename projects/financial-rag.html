Problem & Approach

Information Overload in Financial Markets

Financial markets generate approximately 10,000+ news articles daily across outlets like Bloomberg, Reuters, the Wall Street Journal, and MarketWatch. Analysts, portfolio managers, and traders need to rapidly synthesize this information to answer questions like "What are the current risks to tech stocks?" or "How is the Fed addressing inflation?"

Traditional approaches are broken: manual research takes 30+ minutes per query, keyword search requires knowing exactly what to look for (5–10 minutes), and professional terminals like Bloomberg or FactSet cost thousands per seat. Generic LLMs hallucinate financial facts and have knowledge cutoffs. News aggregators can surface articles but can't synthesize or reason across them.

RAG was chosen over fine-tuning an LLM on financial data: fine-tuning requires expensive compute and goes stale within days as news changes, while RAG grounds every answer in a live, attributed knowledge base that can be updated without retraining. The key constraint shaping every decision was latency — a system that takes 30 seconds isn't useful in fast-moving markets, which drove the choice of a lightweight embedding model (MiniLM over larger alternatives), a curated 50-article corpus over an open-ended index, and a cloud-hosted inference API over a locally-quantized model.

Key Component Decisions

  • all-MiniLM-L6-v2 for embeddings — fast CPU inference (~10ms/chunk) with strong semantic similarity at only 384 dimensions. Larger models added latency without meaningful precision gains at this corpus size.
  • ChromaDB for vector storage — persistent, local, supports cosine similarity with metadata filtering. ~15 MB index for 120 chunks; no external dependency required.
  • FinBERT (ProsusAI) for sentiment — domain-specific model trained on financial texts. 86% accuracy vs ~70% for generic BERT; "beat expectations" and similar financial language requires domain-aware signals.
  • Dual LLM backends — Llama-3.1-8B-Instruct via HuggingFace Inference API (2–4s) for production; Phi-3 Mini via Ollama (5–10s) for local dev. Switching is a single environment variable.
  • LangChain orchestration — structured prompt templates, context assembly, source attribution, and streaming token output. Reduced boilerplate for prompt construction and chain composition.
Data & Indexing

Building the Knowledge Base

Financial news is sourced from the Finnhub API, covering general market news and company-specific articles for 50 priority tickers across 6 sectors: Technology 25% (AAPL, MSFT, NVDA, META, GOOGL), Financial 25% (JPM, BAC, GS, V, MA), General Market 20% (SPY, Fed policy, macro news), Healthcare 12%, Consumer 10%, Energy 8%.

Raw articles go through a 5-stage filtering pipeline that reduces 500+ raw articles down to 50 high-quality, diverse articles (90% noise reduction):

Stage Filter Logic Remaining
1. Quality Min 100-char content, 15-char title; blocks PR wires & spam keywords ~287
2. Relevance Retains articles mentioning priority tickers or macro topics ~156
3. Deduplication Removes exact & fuzzy duplicates (SequenceMatcher, threshold=0.75) ~98
4. Quality Score Ranks by recency, length, source tier; premium sources +30 pts ~98
5. Diversity Sector quotas enforced (e.g., max 25 Technology articles) 50

Each article is embedded using all-MiniLM-L6-v2 (384-dimensional), split into chunks of 1,000 characters with 100-character overlap via RecursiveCharacterTextSplitter, yielding ~120 chunks. Each chunk retains metadata: title, source, date, URL, FinBERT sentiment label/score, and sector. The index is stored in ChromaDB (~15 MB).

Evaluation

Measuring What Matters

Rather than relying on subjective evaluation, the system was tested rigorously across retrieval quality, generation quality, and latency. An A/B test on article window size (25, 50, 75, 100 articles) showed that 50 articles provided the optimal balance — below 25, precision dropped for niche sector queries; above 50, retrieval precision degraded as irrelevant chunks diluted results.

Latency Breakdown

Operation Latency
Query embedding (MiniLM) 10–20 ms
Vector search (K=5, ChromaDB) 30–50 ms
LLM generation (500 tokens, Llama-3.1-8B) 2–4 sec
Total end-to-end 2.5–4.5 sec

Sentiment validation: 50 articles manually labeled by human annotators. FinBERT achieved 86% overall agreement, 88% positive/negative precision, 82% neutral recall. Neutral class is hardest — financial news often has mixed signals, and article-level sentiment labels don't always reflect query-specific sentiment.

Compared to manual research (30+ min) or Google Search (5–10 min), the system reduces query time by 95%+. The live demo is publicly accessible on Hugging Face Spaces.

IF I WERE TO REVISIT IT

What Scales, What Doesn't

The core architecture holds up well for a single-user research tool. The main limitations stem from the static knowledge base and the 50-article corpus ceiling — both defensible given the latency constraints, but real bottlenecks if the use case scales to multi-user or real-time requirements.

What scales well

  • Multi-stage filtering: the 90% noise reduction from 500+ to 50 articles was the single highest-leverage decision. More isn't better in RAG — a curated, high-quality corpus consistently outperforms a large noisy one at this scale.
  • Domain-specific sentiment model: FinBERT's 86% vs ~70% for generic BERT isn't marginal — financial language has domain-specific signals ("beat expectations" = positive, "missed guidance" = negative) that generic models misclassify systematically.
  • Dual LLM backends: supporting Ollama locally and HuggingFace cloud via a single environment variable meant fast iteration during development without burning API credits. The pattern generalizes well.
  • Lightweight embedding model: MiniLM at 384 dimensions was the right call at this corpus size. Larger embedding models added 5–10× inference time for negligible precision gain on a 120-chunk index.

What doesn't

  • Static knowledge base: the system requires periodic re-ingestion for fresh news. Real-time updates would require webhook integration or a continuous ingestion pipeline — a natural next step that would make the tool genuinely useful for time-sensitive queries.
  • Manual evaluation ceiling: 20–50 manually labeled samples is enough to identify weak points (neutral recall) but not enough to detect rare failure modes. A larger automated evaluation harness with ground-truth QA pairs would harden confidence in the 90% accuracy claim.
  • Neutral sentiment recall: 82% recall on neutral sentiment is the known weak point. Article-level FinBERT labels don't always reflect query-specific sentiment — a chunk-level sentiment approach would be more precise.
  • Single-tenant Spaces deployment: Hugging Face Spaces works well for a public demo; at production load it would need a proper API backend with request queuing and rate limiting.
